# Sparkle 2 GitHub 分发教程

这份教程记录 Selenophile 使用 GitHub Releases、GitHub Pages 和 Sparkle 2 分发更新的流程。

## 当前已经接入的内容

- Sparkle 2 已经通过 Swift Package Manager 链接到 `Selenophile` app target。
- `Project.swift` 定义了当前发布用构建设置：
  - `SPARKLE_FEED_URL = https://kare-udon.github.io/Selenophile/selenophile/appcast.xml`
  - `SPARKLE_PUBLIC_ED_KEY = E/nueJJiuhX7I3zRZEjCtYn50JepeMIbY1LltoIs6rA=`
- `Scripts/package_app.sh` 也可以通过环境变量接收同样的值。
- Sparkle 启动逻辑已经在代码中做了保护。只有 app bundle 里存在真实 HTTPS `SUFeedURL`，并且 `SUPublicEDKey` 非空时，更新检查才会启用。
- 默认通过 `SUEnableAutomaticChecks = true` 和 `SUScheduledCheckInterval = 86400` 启用自动检查；Selenophile 打开时会做一次后台检查，之后按每天一次的周期由 Sparkle 调度。Settings 的 General 页提供检查状态、自动检查开关和手动检查入口。

不要恢复占位 feed URL。代码会有意把 `example.com` 或未替换构建变量视为未启用状态。

## 推荐的 GitHub 布局

使用 GitHub Releases 托管不可变的 release 文件，用 GitHub Pages 托管 appcast：

- Appcast: `https://<owner>.github.io/<pages-repo>/selenophile/appcast.xml`
- 发布说明: `https://<owner>.github.io/<pages-repo>/selenophile/releases/<version>.html`
- DMG 资源: `https://github.com/<owner>/<repo>/releases/download/v<version>/Selenophile-<version>.dmg`

GitHub Releases 可以托管 DMG，但体积很小的静态 appcast 和发布说明通常更适合放在 GitHub Pages。

## 一次性 Sparkle 密钥设置

当 SwiftPM 或 Xcode 已经解析出 Sparkle 后，先定位 Sparkle 的工具：

```bash
find .build ~/Library/Developer/Xcode/DerivedData -path '*/Sparkle/bin/generate_keys' -type f 2>/dev/null
```

运行一次 `generate_keys`：

```bash
/path/to/Sparkle/bin/generate_keys
```

把输出中的 public key 写入 app 配置：

- Tuist/Xcode bundle 路径：在 `Project.swift` 中设置 `SPARKLE_PUBLIC_ED_KEY`。
- SwiftPM fallback 打包路径：把 `SPARKLE_PUBLIC_ED_KEY` 传给 `Scripts/package_app.sh` 或 `Scripts/build_dmg.sh`。

私钥应保存在 macOS Keychain 或加密备份中。不要把私钥提交到 git，不要上传到 GitHub Releases，不要放进 Pages，也不要放在托管 appcast 的同一个公共主机上。

## 配置 Feed URL

当 GitHub Pages URL 准备好后，在 `Project.swift` 中替换占位值：

```swift
"SPARKLE_FEED_URL": "https://<owner>.github.io/<pages-repo>/selenophile/appcast.xml",
"SPARKLE_PUBLIC_ED_KEY": "<public key printed by generate_keys>",
```

如果使用 SwiftPM fallback 打包脚本：

```bash
SPARKLE_FEED_URL="https://<owner>.github.io/<pages-repo>/selenophile/appcast.xml" \
SPARKLE_PUBLIC_ED_KEY="<public key printed by generate_keys>" \
./Scripts/build_dmg.sh
```

## 构建发布 DMG

每次发布前，明确更新 `version.env`：

```bash
MARKETING_VERSION=0.1.1
BUILD_NUMBER=2
```

`MARKETING_VERSION` 是用户可见版本。`BUILD_NUMBER` 会映射到 `CFBundleVersion`，并且必须单调递增；Sparkle 会用它判断一个更新是否比当前版本更新。

构建 DMG：

```bash
SPARKLE_FEED_URL="https://<owner>.github.io/<pages-repo>/selenophile/appcast.xml" \
SPARKLE_PUBLIC_ED_KEY="<public key>" \
./Scripts/build_dmg.sh
```

默认输出为 `dist/Selenophile-<MARKETING_VERSION>.dmg`。

如果要进行真正的公开分发，优先使用 Developer ID 签名并完成 notarization。当前本地脚本默认使用 ad-hoc signing，这对开发很方便，但不等同于已经 notarized 的生产更新链路。

## 生成或更新 Appcast

创建一个同时包含 release archive 和对应发布说明的目录：

```bash
mkdir -p dist/sparkle-updates
cp "dist/Selenophile-${MARKETING_VERSION}.dmg" dist/sparkle-updates/
cp docs/releases/example-release-notes.html "dist/sparkle-updates/Selenophile-${MARKETING_VERSION}.html"
```

运行 Sparkle 的 appcast 生成工具：

```bash
/path/to/Sparkle/bin/generate_appcast dist/sparkle-updates
```

这个工具会给 DMG 签名，生成 `sparkle:edSignature`，并且在目录中存在旧 archive 时生成 delta updates。如果之后手动修改 appcast 中指向 GitHub Releases 的 URL，必须认真验证结果，并且不要在不重新生成签名的情况下改动 DMG 字节内容。

如果你选择手动维护 XML，至少需要包含这些内容：

```xml
<item>
    <title>Version 0.1.1</title>
    <sparkle:version>2</sparkle:version>
    <sparkle:shortVersionString>0.1.1</sparkle:shortVersionString>
    <sparkle:minimumSystemVersion>14.0.0</sparkle:minimumSystemVersion>
    <sparkle:releaseNotesLink>
        https://<owner>.github.io/<pages-repo>/selenophile/releases/0.1.1.html
    </sparkle:releaseNotesLink>
    <pubDate>Fri, 08 May 2026 12:00:00 +0000</pubDate>
    <enclosure
        url="https://github.com/<owner>/<repo>/releases/download/v0.1.1/Selenophile-0.1.1.dmg"
        sparkle:edSignature="<generated signature>"
        length="<generated byte length>"
        type="application/octet-stream" />
</item>
```

优先使用 `generate_appcast`，不要手写 XML。原因是它会签名 archive，并确保签名元数据和文件的真实字节内容一致。

## 发布到 GitHub

1. 创建一个 GitHub Release，例如 `v0.1.1`。
2. 上传 `Selenophile-0.1.1.dmg` 到这个 release。
3. 把 `appcast.xml`、发布说明和所有生成的 `.delta` 文件发布到 GitHub Pages 路径。
4. 确认 `appcast.xml` 中的 URL 都能通过 HTTPS 访问。
5. 发布后不要替换 release asset。如果 DMG 内容发生变化，就递增 `BUILD_NUMBER`、重新构建、重新签名，然后发布一个新版本。

## 本地测试清单

安装一个旧版本 build，并托管一个指向新版本 build 的 appcast：

```bash
defaults delete com.udon.selenophile SULastCheckTime 2>/dev/null || true
./run-menubar.sh
```

打开 Settings，手动运行更新检查。Sparkle 会把详细失败信息记录到 Console.app。

如果要做接近生产环境的测试，请使用 Developer ID 签名并 notarized 的 build。部分 Sparkle 签名和安装行为无法用开发签名或 ad-hoc 签名完整验证。
