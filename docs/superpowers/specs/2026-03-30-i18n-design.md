# Selenophile i18n 设计

## 背景

当前 Selenophile 的界面文案大多直接写在 SwiftUI 视图和配置常量中，语言固定为中文。这会带来两个问题：

- 应用无法根据用户系统语言自动匹配界面语言
- 用户无法在设置中手动切换到自己偏好的语言

本次目标是在不重构整体架构的前提下，为应用增加一套原生、低风险、可持续扩展的国际化方案。

## 目标

- 应用首次启动时默认跟随系统语言
- 支持用户在设置页手动指定界面语言
- 手动设置后，界面尽量即时刷新，不要求重启
- 支持前 20 个常用语言
- 未命中支持语言时，稳定回退到英文
- 新方案同时覆盖菜单栏面板、设置窗口、日志窗口和可共享的应用级标题文案

## 非目标

- 不在本次引入机器翻译服务或在线语言包下载
- 不覆盖前 20 语言之外的额外语言变体
- 不对所有历史代码做大规模重构
- 不在本次实现中扩展复杂的日期、数字、单位本地化格式策略

## 支持语言范围

第一版支持以下 20 种语言：

- 英语 `en`
- 简体中文 `zh-Hans`
- 西班牙语 `es`
- 印地语 `hi`
- 阿拉伯语 `ar`
- 葡萄牙语（巴西）`pt-BR`
- 孟加拉语 `bn`
- 俄语 `ru`
- 日语 `ja`
- 旁遮普语 `pa`
- 德语 `de`
- 爪哇语 `jv`
- 韩语 `ko`
- 法语 `fr`
- 泰卢固语 `te`
- 马拉地语 `mr`
- 土耳其语 `tr`
- 泰米尔语 `ta`
- 越南语 `vi`
- 乌尔都语 `ur`

选择原则是优先覆盖常见桌面用户语言，并控制第一版翻译与测试规模。

## 方案选择

比较过的方案：

1. 原生 `Localizable.strings` + 应用内 locale 覆盖
2. 自建字典式翻译服务
3. 原生与自建混合

选择方案 1。

原因：

- 最符合 Apple 平台惯例
- 可以直接服务 SwiftUI、AppKit 和未来 Widget 文案
- 维护成本最低
- 后续增加语言时只需补充语言资源，不需要改运行时架构

## 架构设计

### 语言模型

新增一个应用语言枚举，用于表达：

- 跟随系统
- 20 个受支持的具体语言

该枚举负责：

- 生成设置页展示名称
- 生成对应 locale 标识
- 根据 `Locale.preferredLanguages` 解析系统首选语言
- 在未命中时回退到英文

解析规则应优先支持语言族兼容，例如：

- `zh-CN`、`zh-SG` 映射到 `zh-Hans`
- `pt` 优先回退到 `pt-BR`
- `en-GB`、`en-US` 映射到 `en`

若系统返回更细分的地区标识，但主语言能匹配到受支持语言，则使用该受支持语言。完全无法匹配时使用英文。

### 配置持久化

在现有配置模型中新增一个界面语言字段。

- 默认值：`system`
- 存储形式：稳定的字符串代码
- 配置读取失败或字段缺失时，回退到 `system`

该字段应进入现有配置读写链路，保持与其他设置一致的持久化方式。

### 运行时语言解析

应用启动时统一计算“当前生效语言”：

- 若用户手动指定语言，则使用手动值
- 若用户选择跟随系统，则读取系统首选语言并映射到支持语言
- 若映射失败，则使用英文

SwiftUI 根视图通过环境注入当前 locale，使依赖 `LocalizedStringKey` 的视图自动刷新。

AppKit 管理的窗口标题等文本不完全依赖 SwiftUI 环境，因此应通过统一的本地化访问层显式读取当前语言对应文案。

### 本地化资源

新增语言资源目录与 `Localizable.strings`。

第一版文案覆盖范围：

- 菜单栏面板
- 设置页
- 日志窗口
- 通用按钮文案
- 配置与空状态提示
- 应用窗口标题

第一版不追求把所有内部调试日志消息都翻译完；如果某些日志更偏技术调试，可继续保留英文或沿用当前语言，但所有用户直接看到的主要界面元素应统一进入本地化资源。

## 设置页交互

设置页新增“界面语言”选择控件。

交互规则：

- 默认显示“跟随系统”
- 列出全部受支持语言的本地名称
- 用户切换后立即保存配置
- 保存成功后，当前设置窗口和菜单栏界面尽量即时切换语言
- 不要求弹出“需要重启”提示

设置页中“跟随系统”一项建议补充当前解析结果，例如：

- 跟随系统（当前：English）
- 跟随系统（当前：简体中文）

若实现上会明显增加复杂度，则第一版可只显示“跟随系统”，不附带当前解析结果。

## 代码改动边界

第一版优先改造这些区域：

- `AppConfig` 中可见标题文案
- `SettingsView`
- `MenuContentView`
- `LogView`
- 其他用户直接看到的固定文案

改造策略是：

- 将硬编码文本替换为本地化 key
- 尽量复用现有结构，不拆大组件
- 仅在需要显式获取字符串的地方增加轻量本地化辅助层

## 错误处理与回退

- 缺失语言配置时回退到 `system`
- 缺失 locale 映射时回退到英文
- 个别 key 在某语言缺失时，由系统回退到基准语言资源
- 本地化读取失败不应阻塞主流程或配置保存

## 测试策略

优先覆盖可稳定单测的逻辑：

- 语言枚举与字符串代码双向转换
- 系统语言解析到受支持语言的映射
- 未命中时回退到英文
- 配置模型读写新增字段后保持兼容

界面层不强求全面 UI 测试，但实现后至少应人工验证：

- 系统语言为受支持语言时默认显示正确
- 系统语言不受支持时默认回退到英文
- 设置页切换语言后，窗口与菜单文案即时刷新
- 重新启动应用后保留用户手动设置

## 翻译策略

由于本程序文案量较小，第一版直接在仓库内维护人工翻译。

翻译原则：

- 优先自然、简短、桌面应用语气
- 避免逐词硬译
- 按现有视觉布局控制长度，减少按钮和卡片溢出风险

## 验收标准

- 应用能根据系统语言自动选择界面语言
- 用户可在设置页手动切换界面语言
- 手动设置会持久化，重启后仍生效
- 不受支持的系统语言会回退到英文
- 菜单栏面板、设置窗口、日志窗口的主要界面文案均完成本地化
- 至少支持 20 种语言
- 新增逻辑具备对应单元测试
